01 어떤 목적으로 만들었나

커뮤니티 여론을 "관측 가능한 데이터"로

커뮤니티 반응은 빠르게 흘러가고 곧 사라집니다. 이 도구는 그 흐름을 주기적으로 수집·축적해, 시간이 지나도 되짚어 볼 수 있는 데이터로 만들고 — 그 위에서 화제·분위기·활동 패턴을 파악하고 특정 주제의 여론을 깊이 있게 정리하는 것을 목표로 합니다.

🗂️축적

지정한 날짜·기간의 글을 본문·댓글까지 빠짐없이 모아 DB로 남긴다. 재수집해도 중복 없이 최신 상태를 유지한다.

🔭관측

언제 활발한지, 오늘 뭐가 터졌는지, 어떤 단어가 도는지를 대시보드에서 즉시 본다.

🧠심층 이해

"이 주제 반응 어때?"라고 물으면 AI가 스스로 관련 글을 찾아 근거와 함께 여론을 정리한다.

🖱️누구나

수집·조회·분석을 코드 없이 웹 UI에서 다룬다.

02 왜 만들었나

흩어지고 사라지는 여론, 손으로는 못 따라간다

Q1
데이터가 휘발된다. 글·댓글은 시간이 지나면 묻히거나 삭제된다. 사후에 "그때 반응이 어땠지?"를 확인할 방법이 없다. → 주기 수집으로 타임캡슐을 만든다.
Q2
눈으로 훑기엔 양이 많다. 하루 수백 글·수천 댓글을 사람이 다 읽을 수 없다. → 통계·시계열·키워드로 큰 그림을 먼저 잡는다.
Q3
은어·맥락 때문에 검색이 안 통한다. 커뮤니티는 표준어 대신 은어를 쓴다("젬이오"를 "젬·젬식"이라 부른다). 단순 검색은 절반을 놓친다. → AI가 실제 쓰는 은어까지 찾아 검색·정리한다.
Q4
인사이트까지 자동화하고 싶다. 수치만으론 "무슨 얘기가 오갔는지"를 모른다. → LLM이 근거 글을 인용하며 여론 리포트를 써준다.

수집 → 저장 → 분석·리포트, 세 층

📥 수집 탭

갤러리·날짜·기간을 지정해 수집을 실행하고 진행 상황·이력을 본다.

📋 글 목록 탭

📊 분석 탭

🧠 리포트 탭

자연어 질문 → AI가 은어까지 검색어를 발굴해 전수 분석한 구조화 리포트.

3.1 스크래핑 알고리즘

날짜 범위를 "정확히" 훑고, 삭제까지 반영한다

핵심은 두 가지 — 범위를 넘치거나 모자라지 않게 걷기, 그리고 재수집 시 추가·수정·삭제를 모두 반영해 항상 원본과 같은 상태를 유지하기.

  1. 목록 걷기 (범위 커버) — 갤러리 목록은 최신순이다. 페이지를 넘기며 지정 기간 [시작, 끝]에 드는 글만 대상에 담는다. 기간보다 새 글은 건너뛰되 멈추지 않고(예: 과거를 수집할 때 오늘 글), 기간보다 오래된 글이 나오는 페이지에 닿으면 "범위를 지났다"고 보고 멈춘다. 공지 글머리는 제외.
  2. 상세 수집 — 각 대상 글의 본문 페이지를 가져와 본문 텍스트·추천/비추·조회수·성인여부를 파싱한다. 성인(NSFW) 글은 로그인이 필요해 본문을 못 받으므로 메타만 저장하고 성인 플래그를 세운다.
  3. 댓글 수집 — 댓글 전용 엔드포인트를 페이지 끝까지 순회해 대댓글 계층까지 모은다.
  4. 멱등 저장 (upsert) — 글은 글번호, 댓글은 (글번호, 댓글번호)가 고유 키. 같은 게 다시 오면 새 행을 쌓지 않고 최신 값으로 덮어쓴다. 그래서 재수집이 곧 "추천·댓글 변동분 업데이트" 역할을 한다.
  5. 삭제 반영 — 재수집 때 사라진 댓글은 DB에서 제거한다. 또 목록 걷기가 기간을 완전히 커버했다면, 살아있는 목록에 없는 저장된 글은 삭제/블라인드로 보고 제거한다. safe 단, 페이지 상한에 걸리거나 일부 실패한 회차에서는 "안 보임 ≠ 삭제"이므로 삭제 스윕을 건너뛴다(오삭제 방지).
⚙
매너 수집. 요청 사이에 랜덤 딜레이를 두고, 실패 시 지수 백오프로 재시도하며, 403/429(차단 신호)를 만나면 쿨다운 후 재개한다. 개별 글 하나가 실패해도 전체 수집은 멈추지 않는다.

3.2 각 분석·리포트의 알고리즘

지표는 정확하게, 리포트는 은어까지

모든 분석은 상단 공통 필터(갤러리·기간·키워드)로 범위를 좁힐 수 있고, 필터가 없으면 수집된 전체가 대상입니다. 키워드 기반 분석은 공통 토크나이저(아래)를 공유합니다.

🧩 공통 토크나이저 — 명사 추출

워드클라우드·이슈 버스트가 공유하는, 문장에서 "의미 있는 단어"만 골라내는 단계다. 한국어는 젬이오가·젬이오는처럼 단어에 조사가 달라붙어, 띄어쓰기로 자르거나 글자를 부분일치시키면 같은 단어를 하나로 못 묶는다. 그래서 형태소 분석기(kiwipiepy)를 쓴다.

# 예: "젬이오가 검열을 너무 세게 한다"
형태소 분석 →  젬이오/명사  가/조사  검열/명사  을/조사  너무/부사  세/형용사  게/어미  하/동사 …
채택       →  젬이오              검열
① 품사품사 태그로 명사만 채택. 형태소 분석기는 문장을 의미 최소 단위로 쪼개고 각 조각에 품사표(명사·조사·동사·형용사·어미…)를 붙인다. 그중 일반명사·고유명사·영문(알파벳) 태그가 붙은 것만 남기고 나머지(조사·동사·어미 등)는 버린다. 그래서 "젬이오가"에서 조사 "가"가 떨어져 젬이오만 깔끔히 남는다 — 단순 부분일치로는 불가능한 일이다.
② 길이2자 미만은 버린다. "것·수·때"처럼 한 글자 명사는 대개 뜻이 옅은 의존명사라 노이즈다. 글자 수가 2 미만인 토큰을 제외한다.
③ 불용어불용어 목록과 대조해 제거. 명사여도 흔하고 밋밋해 변별력이 없는 단어(예: 사람·생각·정도)와 본문에 섞여 든 CSS 잔재(px·rem·div 등)를 미리 불용어 사전(파일)에 등록해 두고, 토큰이 그 목록에 있으면 뺀다. 영문은 소문자로 통일해 GPT·gpt를 같은 단어로 센다.

이 앞에서, 본문에 텍스트로 섞여 든 인라인 HTML/CSS(<div>·px·font 등 — AI채팅 갤러리에선 상태창 템플릿을 본문에 붙여넣는다)도 함께 걷어낸다. 이 정리는 분석할 때만 적용되고 원본 글은 건드리지 않는다.

📇A. 개요 지표

무엇을현재 필터 범위의 한눈 요약.
어떻게글 수 · 댓글 수 · 고유 작성자 수 · 평균 댓글/조회 · 성인글 수를 단순 집계. 가중치·모델 없음.

🔥B. 이슈 버스트 — 급상승 & 신규 등장

무엇을"오늘 갑자기 튀어나온 단어"를 찾는다.
어떻게TF-IDF가 아니라 시간축 비중 비교다. 각 글 제목+본문에서 명사를 날짜별로 센 뒤, 대상일에 그 단어가 차지하는 비중을 과거 며칠의 비중과 비교한다.
버스트 = (오늘 그 단어 비중) ÷ (과거 비중 · 라플라스 스무딩)
  • 급상승 = 이 비율이 높은 순. 신규 등장 = 과거엔 아예 없다가 오늘 나온 단어(빈도순).
  • 스무딩(＋보정)으로 "과거에 없던 단어가 무한대로 튀는" 것을 막아 유한한 점수를 준다.
!
문턱이 "하루 2회 이상"뿐이라, 긴 글 하나가 어떤 단어를 두 번 언급해도 후보가 된다. "몇 개의 서로 다른 글에서 나왔는지"는 아직 보지 않아 한 글 편중에 취약하다(개선 여지). 댓글은 연도 정보가 없어 제외.

🧠F. 심층 리포트 AI 에이전트

가장 핵심적인 기능. 자연어 질문을 받아, AI가 스스로 검색어를 찾아가며 관련 글을 전수 분석하고 근거를 인용한 구조화 리포트를 만든다. 단순 키워드 검색이 못 잡는 은어를 극복하는 게 설계의 핵심이다.

  1. 검색어 발굴 (은어 확장) — AI가 질문의 표준어로 먼저 검색한 뒤, 검색 결과의 실제 표현(특히 댓글)에서 커뮤니티가 쓰는 은어·동의어·표기변형을 찾아낸다. 예: "젬이오" → 젬·젬식·Gemini·눅눅(눅눅한 출력)·급발진(캐릭터 붕괴). 정보검색의 의사 적합성 피드백(PRF) 기법이다.
  2. 전수 수집 — 발굴한 검색어로 매칭되는 글을 (성인글 제외) 폭넓게 모아, 각 글의 제목·본문·댓글을 분석용 문서로 만든다.
  3. Map-Reduce 요약 — 문서가 많으면 여러 묶음으로 나눠 각각 요약(map)한 뒤, 부분 요약들을 하나로 종합(reduce)한다. 근거가 된 글 번호는 합쳐서 보존한다.
  4. 구조화 리포트 — 개요 · 분위기 · 주요 주제 · 긍정 · 부정 · 쟁점 · 대표 반응으로 정리하고, 각 항목마다 근거 글 링크를 붙여 원문으로 검증할 수 있게 한다.
  5. 기록 — 질문·리포트·근거·검색어를 DB에 남겨 히스토리로 되짚어 본다.
# 파이프라인 한눈에
질문  "젬이오 검열 반응 어때?"
  │
  ├─ ① 에이전트 검색·은어 발굴  → [젬이오, 젬, 젬식, 눅눅, 급발진, 검열]
  ├─ ② 전수 수집              → 매칭 306글 → 상위 60 분석
  ├─ ③ map-reduce 요약
  └─ ④ 구조화 리포트 + 근거링크  → 개요·긍정·부정·쟁점·인용
  ↳ DB 로깅

🗣️ 어떤 프롬프트를 쓰나 — 역할이 다른 두 개

리포트는 성격이 다른 두 프롬프트로 나뉜다: ①검색어를 찾는 에이전트와 ②여론을 분석하는 분석가. 한 프롬프트에 다 시키지 않고 역할을 분리해 각자 한 가지만 잘하게 한다.

1단계 검색어 발굴 에이전트

  • 역할 고정 — "질문에 직접 답하지 말고, 조사에 필요한 검색어를 찾는 것이 임무다"
  • 은어 지시 — "커뮤니티는 표준어 대신 은어를 쓴다. 검색 결과의 제목·본문·댓글에서 같은 대상을 가리키는 실제 은어·동의어·표기변형을 찾아내라"
  • 반복 유도 — "1차 결과가 적거나 비어도 부분 힌트에서 은어를 유추해 재검색하라"
  • 출력 계약 — "설명 없이 검색 키워드 JSON 배열만 반환. 예: ["젬이오","젬","gemini"]"

2단계 여론 분석가 (map·reduce)

  • 역할 고정 — "너는 한국어 커뮤니티 여론 분석가다. 슬랭·은어·반어를 감안해 무슨 이야기를 하고 어떻게 평가하는지 분석하라"
  • 고정 스키마 — 개요·분위기·주제·긍정·부정·쟁점·인용을 정해진 JSON 형식으로만 반환하도록 강제
  • 근거 규칙 — "모든 항목에 판단 근거가 된 글 번호(post_nos)를 반드시 포함하라. 근거 없는 항목은 만들지 마라"
  • 종합(reduce) — "여러 묶음의 부분 분석을 하나로 합치되, 각 항목의 근거 글 번호는 합집합으로 모두 보존하라"

여기에 더해, 결과가 프로그램에서 바로 쓰이도록 형식 규칙을 항상 덧붙인다 — "유효한 JSON 하나만, 코드펜스 금지, 값 안에 줄바꿈 금지, 인용부호는 큰따옴표 대신 작은따옴표". 그래도 깨지면 1회 자동 복구(JSON 고치기)를 시도한다.

◆
이 방식은 무거운 에이전트 프레임워크(LangGraph)나 지식그래프(GraphRAG) 없이 얇은 도구호출 루프로 구현해, 의존성을 늘리지 않으면서 "스스로 검색해 답하는" 동작을 얻는다. 검색은 현재 키워드 부분일치 기반이며, 동의어·표현 다양성을 더 넓히려면 임베딩 기반 의미검색이 향후 후보다.
